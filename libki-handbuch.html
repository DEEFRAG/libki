1. Einleitung

1.1. Was ist Libki?

Libki ist ein freies, quelloffenes, plattformübergreifendes System zur Reservierung und Zeitverwaltung von PCs, mit dem sich zeitlich begrenzter Zugang zu Computern in einem Netzwerk realisieren lässt. Libki eignet sich ideal für Umgebungen, in denen eine kontrollierte Computernutzung entscheidend ist – etwa öffentliche Zugangssysteme, Bibliotheken, Schulcomputerräume und mehr! Es besteht aus zwei Hauptbestandteilen: dem Libki-Server und dem Libki-Client.

Libki kann außerdem als Druckverwaltungsdienst dienen, sodass Benutzer sowohl von Libki-gesteuerten Clients als auch per Datei-Upload auf den Libki-Server drucken können.

Weitere Informationen zu Libki finden Sie auf der offiziellen Libki-Website libki.org.

1.1.1. Der Libki-Server

Der Libki-Server nutzt ein webbasiertes Administrationssystem, das von jedem vernetzten PC aus über einen Webbrowser aufgerufen werden kann. Über die Web-Administration lassen sich Benutzer anlegen und löschen, das verbleibende Zeitguthaben eines Kontos ändern, Nachrichten an Benutzer senden oder hinterlassen, Benutzer abmelden und/oder sperren sowie einsehen, welche Client-Computer verfügbar und welche gerade in Benutzung sind. Zusätzlich verfügt das System über eine öffentliche Weboberfläche, über die eingesehen werden kann, welche Kiosk-PCs verfügbar bzw. nicht verfügbar sind, und über die Reservierungen vorgenommen werden können.

1.1.2. Der Libki-Client

Der Libki-Client ist plattformübergreifend einsetzbar und läuft auf zahlreichen Betriebssystemen, darunter Microsoft Windows und Linux. Der Libki-Client zeigt einen Zähler an, der dem Benutzer mitteilt, wie viele Minuten noch verbleiben, und bietet die Möglichkeit, sich vorzeitig abzumelden. Ungenutzte Minuten können an jedem anderen Computer verwendet werden, auf dem der Libki-Client läuft. Der Libki-Client kann so eingestellt werden, dass er sich beim Abmelden vom Betriebssystem abmeldet oder den Computer sogar neu startet, sobald sich ein Benutzer vom Libki-Client abmeldet.

1.2. Empfehlungen für den Libki-Server

Zum Zeitpunkt der Erstellung dieses Handbuchs wird empfohlen, den Libki-Server unter Debian Linux, Ubuntu Server oder mittels Docker zu betreiben.

Anleitungen zur Bereitstellung finden Sie im Abschnitt Installation.

Der Libki-Server ist für alle gängigen Webbrowser ausgelegt, wird jedoch hauptsächlich mit Chrome und Firefox getestet.

2. Installation

2.1. Automatische Installation in einem Schritt

Möchten Sie Libki so schnell wie möglich einrichten und verfügen über einen dedizierten Server? Dann ist dies die richtige Wahl für Sie. Stellen Sie einfach sicher, dass auf Ihrem Server Ubuntu 18.04, 20.04 oder Debian 11 (Bullseye) läuft und dass kein Benutzer namens „libki“ existiert.

wget -O- install.libki.org | bash

(Übrigens handelt es sich dabei um den Buchstaben O, nicht die Ziffer 0.)

2.2. Alternative Methode

Nicht jeder möchte ein Skript direkt aus dem Web per Pipe in bash leiten. Das verhindert nämlich, dass Sie den Code lesen können, den Sie gleich auf Ihrem System ausführen. Wenn Sie ihn vorher durchsehen möchten, ist diese Alternative das Richtige für Sie:

git clone https://github.com/libki/libki-server.git
cd libki-server
bash install.sh

2.3. Docker

Das offizielle Libki-Server-Docker-Image muss mit einem Datenbankserver verbunden werden – es enthält selbst keine MySQL-Instanz. Es gilt als bewährte Praxis, einen MySQL- oder MariaDB-Container im selben Docker-Netzwerk zu verwenden.

Das Libki-Server-Image ist auf Docker Hub verfügbar.

Dieser Container erfordert das Setzen einiger Umgebungsvariablen, damit er ordnungsgemäß funktioniert. Im Folgenden ein Beispiel für die benötigten Umgebungsvariablen:

LIBKI_INSTANCE=<INSTANCE> (1)
LIBKI_DB_DSN=dbi:mysql:<DB_NAME>;host=<DB_HOSTNAME>;port=3306
LIBKI_DB_USER=<DB_USERNAME>
LIBKI_DB_PASSWORD=<DB_PASSWORD>
LIBKI_DB_HOST=<DB_HOSTNAME>
LIBKI_DB_PORT=3306
LIBKI_DB_DATABASE=<DB_NAME>
LIBKI_TZ=America/Denver (2)
1Instanzname; kann bei Einzelinstanz-Servern leer bleiben
2Ihre Zeitzone finden Sie in der Dokumentation hier.

Unter docker/docker-compose.yml wird eine Docker-Compose-Datei bereitgestellt, mit der sowohl der Libki-Server als auch ein MariaDB-Container gestartet werden. Umgebungsvariablen sollten unter docker/.env gespeichert werden.

2.3.1. Cronjobs

Die Libki-Cronjobs sind nicht so eingerichtet, dass sie innerhalb des Docker-Containers laufen. Stattdessen sollten die Cronjobs auf dem Host-Rechner eingerichtet werden.

Fügen Sie Folgendes zur Crontab des root-Benutzers oder zu einer Datei in /etc/cron.d/ hinzu.

* * * * * root docker exec <container-name> /app/script/cronjobs/libki.pl
0 0 * * * root docker exec <container-name> /app/script/cronjobs/libki_nightly.pl

2.4. Libki-Server mit TLS/SSL verwenden

Der Libki-Server bietet keine native SSL-Unterstützung. Um SSL mit Libki zu nutzen, sollten Sie einen Reverse-Proxy mit SSL-Terminierung verwenden, wie z. B. Apache, nginx, Traefik usw. Standardmäßig verwendet Libki Apache als Proxy.

Sobald SSL eingerichtet ist, muss der Proxy den benutzerdefinierten HTTP-Header X-Request-Base mit dem Wert Ihrer Server-URL unter Verwendung von SSL senden. Lautet Ihr FQDN beispielsweise libki.mylib.org, setzen Sie X-Request-Base auf https://libki.mylib.org.

3. Konfiguration

3.1. Einstellungen

3.1.1. Zeiteinstellungen

Standardzeitkontingent pro Tag

Diese Einstellung legt die Gesamtzahl der Minuten fest, die einem Benutzer standardmäßig pro Tag zur Verfügung stehen.

HinweisDer Wert von Standardzeitkontingent pro Tag sollte eine ganze Zahl sein.
Standardzeitkontingent pro Sitzung

Diese Einstellung legt die Gesamtzahl der Minuten fest, die einem Benutzer pro Sitzung und Tag zur Verfügung stehen. Ist ein System also mit 120 Minuten Zeitkontingent pro Tag und 30 Minuten Zeitkontingent pro Sitzung konfiguriert, kann sich ein Benutzer viermal täglich für jeweils 30 Minuten anmelden.

HinweisDer Wert von Standardzeitkontingent pro Sitzung sollte eine ganze Zahl sein.
Standard-Gastzeitkontingent pro Tag

Entspricht dem Standardzeitkontingent pro Tag, gilt jedoch für Gastkonten anstelle regulärer Benutzer.

HinweisDer Wert von Standardzeitkontingent pro Tag sollte eine ganze Zahl sein.
Standard-Gastzeitkontingent pro Sitzung

Entspricht dem Standardzeitkontingent pro Sitzung, gilt jedoch für Gastkonten anstelle regulärer Benutzer.

HinweisDer Wert von Standard-Gastzeitkontingent pro Tag sollte eine ganze Zahl sein.

3.1.2. Automatische Zeitverlängerung

Libki verfügt über ein automatisches Zeitverlängerungssystem, das es Benutzern ermöglicht, über die vorab festgelegte Anzahl an Minuten pro Sitzung hinaus an einem Client-Computer angemeldet zu bleiben.

Verlängerungsdauer

Die Zeitspanne (in Minuten), um die die Sitzungszeit eines Benutzers automatisch verlängert wird.

HinweisDer Wert von Verlängerungsdauer sollte eine ganze Zahl sein.
Zeit verlängern bei

Diese Einstellung steuert, zu welchem Zeitpunkt eine Verlängerung ausgelöst wird. Eine Sitzung wird verlängert, wenn die verbleibende Sitzungszeit des Benutzers unter diesen Wert (in Minuten) fällt.

HinweisDer Wert von Zeit verlängern bei sollte eine ganze Zahl sein.
Zeit verlängern, außer wenn

Diese Einstellung legt fest, ob ein Benutzer für eine automatische Zeitverlängerung infrage kommt. Es gibt zwei Optionen:

  • Client des Benutzers ist reserviert – Diese Option begrenzt eine Zeitverlängerung so, dass sie sich nicht mit einer bestehenden Reservierung des Client-Computers des Benutzers überschneidet. Reservierungen für andere Clients werden nicht berücksichtigt. Der Reservierungsabstand wird dabei berücksichtigt.
  • Ein beliebiger Client ist reserviert – Diese Option verhindert eine Zeitverlängerung, wenn irgendein Client aktuell reserviert ist. Ist irgendein Client reserviert, findet keine Zeitverlängerung statt, selbst wenn der Client des Benutzers selbst nicht reserviert ist.
Beim Verlängern der Zeit

Diese Einstellung bestimmt, wie Minuten dem Konto eines Nutzers gutgeschrieben werden, wenn eine automatische Zeitverlängerung erfolgt. Es gibt zwei Optionen:

  • Minuten vom Tageskontingent abziehen – Diese Option verschiebt Minuten vom Tageskontingent des Benutzers in die Sitzungsminuten. Das bedeutet, der Benutzer kann den Client-Computer weiter nutzen, jedoch nur bis zu seinem täglichen Zeitkontingent.
  • Keine Minuten vom Tageskontingent abziehen – Diese Option fügt der Sitzung eines Benutzers Minuten „aus dem Nichts“ hinzu. Dadurch wird nicht beeinflusst, wie viele Sitzungen ein Benutzer pro Tag hat.

3.1.3. Client-Verhalten

Client-Reservierungen

Diese Einstellung steuert, wie Clients von Benutzern genutzt werden können. Es gibt drei Möglichkeiten:

  • Wer zuerst kommt, mahlt zuerst. Erlaubt einem Benutzer, sich an jedem gerade nicht genutzten Client anzumelden.
  • Nur mit Reservierung. Erfordert, dass ein Benutzer eine Reservierung für einen Client vornimmt, bevor er sich anmeldet.
  • Wer zuerst kommt, mit Reservierungsoption. Ein hybrider Ansatz, der es Benutzern erlaubt, sich an jedem Client anzumelden, der weder genutzt noch reserviert ist.
HinweisDie Einstellung „Benutzernamen anzeigen“ steuert, ob der Benutzername auf dem reservierten Client-Computer angezeigt wird.
Client-Inaktivität

Diese Einstellungen steuern die standardmäßige automatische Abmeldung aufgrund von Benutzerinaktivität.

HinweisDiese Servereinstellungen können pro Client überschrieben werden. Weitere Details finden Sie unter Client-Konfiguration.
Warnung bei Client-Inaktivität

Warnt den Benutzer, dass er nach dieser Anzahl an Minuten Inaktivität automatisch abgemeldet wird.

Abmeldung bei Client-Inaktivität

Meldet den Benutzer nach dieser Anzahl an Minuten Inaktivität automatisch ab.

Verzögerungsgrenze für Client-Registrierungsaktualisierung

Wenn ein Libki-Client gestartet wird, verbindet er sich regelmäßig mit dem Libki-Server, um sich selbst zu registrieren. Registriert sich ein Client nicht innerhalb dieser Anzahl an Minuten erneut, entfernt der Libki-Server ihn aus der Liste der aktiven Clients.

HinweisDer Wert von Verzögerungsgrenze für Client-Registrierungsaktualisierung sollte eine ganze Zahl sein.
Client-Login-Banner

Die Client-Banner sind optionale Bereiche oben und unten auf dem Login-Bildschirm des Libki-Clients. Sie funktionieren ähnlich wie Webbrowser. Daher ist alles, was in einem Webbrowser angezeigt werden kann, auch in den Bannerbereichen anzeigbar (sofern die Größe es zulässt).

Quell-URL

Die URL für das Bild oder HTML, das Sie im Bannerbereich anzeigen möchten.

Breite

Wenn die Quell-URL ein Bild ist, kann eine bestimmte Breite erzwungen werden, anstatt die tatsächliche Bildbreite zu verwenden. Leer lassen, um die tatsächliche Bildbreite zu verwenden.

Höhe

Entspricht der Einstellung Breite für die Quell-URL, jedoch für die Höhe.

3.1.4. Client-Energieoptionen

Einstellungen zum Ausschalten des Clients
Ausschaltaktion

Legen Sie fest, ob die Schaltfläche „Ausschalten“ und die Funktion „automatisches Herunterfahren“ dazu führen, dass die Clients vollständig heruntergefahren oder in einen Ruhezustand versetzt werden.

HinweisBeachten Sie, dass manche Konfigurationen die gleichzeitige Nutzung von Wake-on-LAN mit der Ausschalten-Option nicht zulassen. In diesem Fall muss der Ruhezustand gewählt werden.
Verzögerung des automatischen Herunterfahrens des Clients

Libki kann nach Schließzeit automatisch ein Ausschaltsignal an die Clients senden. Mit dieser Einstellung wird gesteuert, nach wie vielen Minuten nach Schließung dies geschieht. Leer lassen, um dies zu deaktivieren.

Einstellungen zum Einschalten des Clients

Libki verwendet Wake-on-LAN, um Clients einzuschalten. Die Geräte müssen Wake-on-LAN-fähig sein und dies muss aktiviert sein. Dies erfordert unter Umständen eine gewisse Konfiguration im Betriebssystem und/oder in den BIOS-Einstellungen des Geräts.

Signal senden über

Standardmäßig wird das Wake-on-LAN-Paket vom Server gesendet. Es ist möglich, die Konfiguration so einzurichten, dass ein laufender Libki-Client das Paket sendet, um die anderen Geräte aufzuwecken.

HinweisDies kann verwendet werden, um Geräte in einem anderen Netzwerk aufzuwecken, ohne von den Clients aus eine öffentliche Adresse ansprechen zu müssen.
Host

Diese Einstellung ist optional. Bleibt sie leer, wird die begrenzte Broadcast-Adresse (255.255.255.255) verwendet. Das sollte ausreichen, wenn sich der Server im selben lokalen Netzwerk wie die Clients befindet.

Um einen anderen Knoten oder ein entferntes Netzwerk anzusprechen, geben Sie die IP-Adresse oder den vollqualifizierten Domainnamen (FQDN) an.

HinweisEs ist möglich, Wake-on-LAN über das Internet zu nutzen, indem Anfragen entweder per Portweiterleitung auf Router-Ebene oder über einen auf einem dedizierten Gerät laufenden Weiterleitungsdienst an die lokale Broadcast-Adresse weitergeleitet werden.
Port

Die Portnummer für das Magic Packet. Diese Einstellung ist optional und verwendet standardmäßig den Discard-Dienst (9).

MAC-Adressen

Dies ist die Liste der physischen Adressen der Netzwerkadapter, die zum Aufwecken der Geräte verwendet werden.

HinweisDas Format ist eine Adresse pro Zeile, wobei jede Adresse entweder keinen Trenner oder : verwendet.
Automatische Einschaltzeit

Libki kann automatisch ein Aufwecksignal an jedes Gerät senden, dessen MAC-Adresse in der Einstellung MAC-Adressen aufgeführt ist. Alle Geräte werden täglich zu dieser Zeit aufgeweckt. Leer lassen, um dies zu deaktivieren.

3.1.6. Datenaufbewahrung

Anonymisierung des Verlaufs

Diese Einstellung steuert, wie lange Benutzer- und Client-Daten in nicht anonymisierter Form aufbewahrt werden. Bleibt diese Einstellung leer, werden die Daten unbegrenzt aufbewahrt.

Bei der Anonymisierung erhält ein bestimmter Benutzer täglich eine neue zufällige ID. Auf diese Weise können Statistiken die Nutzung weiterhin pro Benutzer und Tag verfolgen, jedoch nicht mehr die Nutzung eines Benutzers über einen längeren Zeitraum hinweg.

HinweisDer Wert von Anonymisierung des Verlaufs sollte eine ganze Zahl sein.
Aufbewahrung des Verlaufs

Diese Einstellung steuert, wie lange (in Tagen) Benutzer- und Client-Daten zum Zweck der Statistikerstellung und Überprüfung des Nutzungsverlaufs aufbewahrt werden. Bleibt diese Einstellung leer, werden die Daten unbegrenzt aufbewahrt.

HinweisDer Wert von Aufbewahrung des Verlaufs sollte eine ganze Zahl sein.
Aufbewahrung inaktiver Benutzer

Diese Einstellung steuert, wie lange (in Tagen) ein Benutzer inaktiv sein kann, bevor er automatisch gelöscht wird. Bleibt diese Einstellung leer, werden alle Benutzer unbegrenzt aufbewahrt.

HinweisDer Wert von Aufbewahrung inaktiver Benutzer sollte eine ganze Zahl sein.

Diese Einstellung sollte bei Bedarf zur Einhaltung der DSGVO genutzt werden.

3.1.7. Gastausweise

Selbstregistrierte Gastausweise

Diese Einstellung steuert, ob Clients über eine Schaltfläche „Als Gast anmelden“ verfügen, die bei der Anmeldung automatisch ein Gastkonto erstellt. Benutzer erhalten eine einmalige Nachricht auf dem Libki-Client mit ihrem Benutzernamen und Passwort, sodass sie ihre Sitzung sperren oder Druckfreigabedienste nutzen können.

HinweisDer Libki-Client muss mindestens einmal neu gestartet werden, bevor diese Option erscheint.
Verbleibende Gastausweiszeit bei Abmeldung verfallen lassen

Ist dies aktiviert, wird jegliche verbleibende Zeit eines Gastkontos auf null gesetzt, nachdem sich dieses Konto von einem Client abgemeldet hat. Dies ist nützlich, um zu verhindern, dass unbefugte Personen verworfene Gastausweise aufnehmen und dadurch zusätzliche Zeit über die von der Bibliotheksrichtlinie erlaubte Zeit hinaus erhalten.

Präfix für Gastausweise

Die Zeichenfolge, mit der jeder Gastausweis-Benutzername beginnen soll. Bleibt sie leer, wird die Zeichenfolge „guest“ verwendet (z. B. guest1, guest2, guest3 usw.).

HinweisDiese Einstellung kann ein Wort oder eine kurze Phrase sein, sollte aber nur Buchstaben und Zahlen enthalten. Vermeiden Sie Leerzeichen oder Sonderzeichen.
Zu erstellende Ausweise pro Stapel

Wird die Schaltfläche Mehrere Gäste verwendet, steuert diese Einstellung, wie viele Gastkonten pro Vorgang erzeugt werden.

HinweisDer Wert von Zu erstellende Ausweise pro Stapel sollte eine ganze Zahl sein.
Bezeichnung Benutzername

Der Text in diesem Feld wird dem Gast-Benutzernamen vorangestellt (z. B. „Benutzername:“).

Bezeichnung Passwort

Diese Einstellung funktioniert wie Bezeichnung Benutzername, jedoch für das generierte Passwort statt für den Benutzernamen.

CSS für Gastausweise

Diese Einstellung ermöglicht es, die gestapelten Gastausweise mit CSS zu gestalten. Der Standardwert für diese Einstellung lautet:

body { /* default body style emulates a pre tag */

3.1.8. ILS-Integration

Benutzer-Hyperlinks

Wenn Sie hier eine URL eintragen, wird der Benutzername in der Benutzertabelle der Web-Administration zu einem Hyperlink, der am Ende den Benutzernamen enthält. Zum Beispiel verweist http://catalog.koha.library/cgi-bin/koha/members/member.pl?quicksearch=1&searchmember= auf die Suchfunktion des Koha-ILS für den jeweiligen Benutzernamen.

SIP-Konfiguration

Single Sign-on mit einem ILS lässt sich über SIP2 realisieren. Einstellungen für den ILS-SIP2-Server können in der Datei libki_local.conf oder in der Einstellung SIP-Konfiguration gespeichert werden.

Um die SIP-Authentifizierung zu aktivieren, müssen Sie Ihre libki_local.conf bearbeiten und einen Abschnitt wie folgt hinzufügen:

<SIP>
    enable 1
    host ils.mylibrary.org
    port 6001
    location LIB
    username libki_sipuser
    password PassW0rd
    terminator CR
    require_sip_auth 0
    enable_split_messages 0
    no_password_check 0 (1)
    fee_limit 5.00 (2)
    deny_on charge_privileges_denied (3)
    deny_on recall_privileges_denied (4)
    deny_on excessive_outstanding_fines (5)
    deny_on_field AB:This is the reason we are denying you (6)
    deny_on_field AF:Your library card has expired.:EXPIRE (7)
    category_field PC (8)
    birthdate_field PB (9)
    pattern_personal_name , (10)
    pattern_personal_name_reverse 1 (11)
    ILS Koha (12)
    skip_import_personal_name 0 (13)
    hold_notification 0 (14)
</SIP>

2Kann entweder ein Gebührenbetrag oder ein SIP2-Feld sein, das die Gebührengrenze definiert (z. B. CC); zum Löschen der Gebührengrenze entfernen
3Sie können SIP2-Benutzerstatus-Flags festlegen, die Benutzern die Anmeldung verweigern

5Die vollständige Liste ist in der SIP2-Protokollspezifikation definiert

7Mit dem Format <Feld>:<Nachricht>:<Wert> wird die Anmeldung verweigert, wenn das Feld <Wert> enthält, und dem Benutzer wird <Nachricht> angezeigt.
8Geben Sie hier ein SIP-Tag an, das die Benutzerkategorie in der SIP-Benutzerabfrageantwort enthält
9Geben Sie hier ein SIP-Tag an, das das Geburtsdatum des Benutzers in der SIP-Benutzerabfrageantwort enthält; Standard ist „PB“. Manche ILS, wie Atriuum von Booksystems, verwenden „PD“.

11Wenn Nachname und Vorname des Benutzers vertauscht sind, korrigiert diese Option die Reihenfolge

  - excessive_outstanding_fines
deny_on_field: "AB:This is the reason we are denying you"
category_field: PC
birthdate_field: PB
pattern_personal_name: ","
ILS: Koha
skip_import_personal_name: 0
hold_notification: 0

Der SIP-Abschnitt erfordert die folgenden Parameter:

  • enable: Auf 1 setzen, um SIP-Authentifizierung zu aktivieren, auf 0, um sie zu deaktivieren.
  • host: Die IP-Adresse oder der FQDN des SIP-Servers.
  • port: Der Port, auf dem der SIP-Server lauscht.
  • location: Der SIP-Standortcode, der zur SIP-Anmeldung passt.
  • username: Der Benutzername des SIP-Kontos, das für Transaktionen verwendet wird.
  • password: Das Passwort des SIP-Kontos, das für Transaktionen verwendet wird.
  • terminator: Je nach SIP-Server entweder CR oder CRLF. Standard ist CR.
  • require_sip_auth: Erfordert dieser SIP-Server eine Message-93-Anmeldung, bevor er verwendet werden kann? Falls ja, sollte dies auf 1 gesetzt und die Felder Benutzername/Passwort ausgefüllt werden. Für Koha sollte dies auf 1 gesetzt sein.
  • enable_split_message: Wenn dieser Server geteilte Nachrichten unterstützt, können Sie dies aktivieren. Für Koha sollte dies auf 0 gesetzt sein.
  • fee_limit: Wie erwähnt kann dies eine feste Zahl oder ein zu prüfendes SIP-Feld sein. Wird die Gebührengrenze überschritten, wird die Benutzeranmeldung verweigert.
  • deny_on: Kann wiederholt werden, um Anmeldungen basierend auf den in der SIP2-Protokollspezifikation beschriebenen Statusflags des Benutzers zu verweigern.
  • deny_on_field: Kann wiederholt werden, um Anmeldungen zu verweigern, wenn das angegebene Feld nicht den Wert „Y“ hat.
  • skip_import_personal_name: Wenn aktiviert, werden Vor- und Nachnamen der Benutzer nicht aus der Benutzerinformationsantwort übernommen und in der Datenbank gespeichert.
  • hold_notification: Wenn auf 0 gesetzt, werden Benachrichtigungen über Vormerkungen deaktiviert.

3.1.15. Erweiterte Einstellungen

Erweiterte Regeln

In diesem Abschnitt können erweiterte Regeln eingerichtet werden, um eine Reihe von Verhaltensweisen anhand verschiedener Kriterien zu bestimmen.

Zum Beispiel:

-
  criteria:
    user_category: MyCategory1
    client_location: MyLocation1
  rules:
    session: 25
    guest_session: 20
-
  criteria:
    client_location: MyLocation1
  rules:
    session: 20
    guest_session: 20
-
  criteria:
    user_category: MyCategory1
  rules:
    session: 15
    guest_session: 20
-
  rules:
    session: 10
    guest_session: 20

Regeln sollten vom Spezifischsten zum Allgemeinsten aufgelistet werden. Fehlt ein Kriterium in einer Regel, wird es beim Abgleich nicht berücksichtigt.

Nehmen wir zum Beispiel an, wir versuchen, uns als Benutzer mit der Kategorie MyCategory1 an einem Client mit dem Standort MyLocation2 anzumelden.

  • Die erste Regel stimmt bei der Kategorie überein, jedoch nicht beim Standort, und wird daher übersprungen.
  • Die zweite Regel prüft nur den Standort, der nicht übereinstimmt, und wird daher übersprungen.
  • Die dritte Regel prüft nur die Kategorie, die übereinstimmt, und wird daher verwendet!
  • Die letzte Regel hat keine Kriterien, sie trifft also auf alles zu.

Gibt es keine solche allgemeingültige Regel, greift Libki auf die Standard-Zeitkontingent-Einstellungen aus dem Abschnitt Zeiteinstellungen zurück.

Selbst wenn eine Regel eigentlich passen würde, wird sie übersprungen, wenn sie nicht die gesuchte „Unterregel“ enthält

Nehmen wir zum Beispiel Folgendes:

-
  criteria:
    client_location: MyLocation1
  rules:
    session: 15
-
  rules:
    session: 10
    guest_session: 20

Angenommen, ein Gast meldet sich an einem Client an einem Standort MyLocation1 an.

  • Die erste Regel wird übersprungen, weil guest_session in dieser Regel nicht definiert ist.
  • Die zweite Regel wird verwendet, weil sie auf alles zutrifft und guest_session tatsächlich definiert.
Regeln können mehrere Werte für ein bestimmtes Kriterium angeben

Zum Beispiel:

-
  criteria:
    user_category:
      - MyCategory1
      - MyCategory2
    client_location: MyLocationA
  rules:
    session: 25
    daily: 50
    guest_session: 20
    guest_daily: 45

Das obige Beispiel würde für jeden Benutzer der Kategorie MyCategory1 oder MyCategory2 gelten, der sich an einem Client am Standort MyLocationA anmeldet.

Das heißt, diese Regel gilt für alle Werte in der Liste dieses Kriteriums.

Man kann es sich auch so vorstellen, dass alle Kriterien UND-verknüpft und alle Werte eines Kriteriums ODER-verknüpft sind.

-
  criteria:
    user_category:
      - Cat1
      - Cat2
    client_location:
      - LocA
      - LocB
  rules:
    session: 25
    daily: 50
    guest_session: 20
    guest_daily: 45

Das würde in SQL etwa so ausgedrückt: SELECT rule WHERE ( user_category = Cat1 OR user_category = Cat2 ) AND ( client_location = LocA OR client_location = LocB )

Regeloptionen

Die Liste aller möglichen Kriterien umfasst:

  • client_location
  • client_name
  • user_category

Die Liste aller möglichen Regeln umfasst:

  • session: Die Anzahl der Minuten pro Sitzung, die einem registrierten Benutzer gewährt wird
  • guest_session: Die Anzahl der Minuten pro Sitzung, die einem Gastbenutzer gewährt wird
  • daily: Das tägliche Minutenkontingent, das einem registrierten Benutzer pro Tag zugewiesen wird
  • guest_daily: Das tägliche Minutenkontingent, das einem Gastbenutzer pro Tag zugewiesen wird
  • no_reservation_required: Erlaubt eine Anmeldung ohne Reservierung, selbst wenn „Client-Reservierungen“ auf „Nur mit Reservierung“ gesetzt ist. Der Wert sollte zum Aktivieren „1“ sein.

3.3. Druckverwaltung

Die Druckverwaltung in Libki kann über zwei verschiedene Backends erfolgen: Libki Print Manager oder CUPS.

3.3.1. Server mit Libki Print Manager konfigurieren

Installieren und konfigurieren Sie Libki Print Manager auf einem Computer, der direkt an Ihre physischen Drucker drucken kann.

Konfiguration der Druckverwaltung einrichten

Nachfolgend ein Beispiel mit 6 virtuellen Druckern, jeweils einer für Schwarzweiß- und Farbvarianten von Simplex, langem Duplex und kurzem Duplex.

---
public_print_release: enabled
printers:
  color:
    chroming: color
    physical_printer_name: somePrinter
    public_printer_name: Color
    plexing: simplex
    type: PrintManager
    cost_per_page: .10
  color-duplex-long:
    chroming: color
    physical_printer_name: somePrinter
    public_printer_name: Color (Duplex/Long)
    plexing: duplexlong
    type: PrintManager
    cost_per_page: .10
  color-duplex-short:
    chroming: color
    physical_printer_name: somePrinter
    public_printer_name: Color (Duplex/Short)
    plexing: duplexshort
    type: PrintManager
    cost_per_page: .10
  monochrome:
    chroming: monochrome
    physical_printer_name: somePrinter
    public_printer_name: Monochrome
    plexing: simplex
    type: PrintManager
    cost_per_page: .05
    auto_release: yes
  monochrome-duplex-long:
    chroming: monochrome
    physical_printer_name: somePrinter
    public_printer_name: Monochrome (Duplex/Long)
    plexing: duplexlong
    type: PrintManager
    cost_per_page: .05
    auto_release: client
  monochrome-duplex-short:
    chroming: monochrome
    public_printer_name: Monochrome (Duplex/Short)
    plexing: duplexshort
    type: PrintManager
    cost_per_page: .05
    auto_release: public
Druckerparameter
  • chroming: Erforderlich, Werte sind color oder monochrome.
  • physical_printer_name: Erforderlich, muss exakt dem Namen des zu verwendenden Windows-Druckers entsprechen.
  • public_printer_name: Erforderlich, so wird der Drucker bei der Druckfreigabe angezeigt.
  • plexing: Erforderlich, Werte sind „simplex“, „duplexlong“ oder „duplexshort“.
  • type: Erforderlich, aktuelle Optionen sind „PrintManager“ oder „cups“.
  • auto_release: Optional
    • bei „yes“ werden Druckaufträge an diesen Drucker nicht zur Freigabe zurückgehalten, sondern automatisch freigegeben, sofern Guthaben verfügbar ist.
    • bei „client“ werden Druckaufträge nur automatisch freigegeben, wenn sie von Libki-Clients stammen (zur Freigabe per Web-Upload zurückgehalten)
    • bei „public“ werden Druckaufträge nur automatisch freigegeben, wenn sie ins Web hochgeladen wurden (zur Freigabe von Libki-Clients zurückgehalten)
  • cost_per_page: der Dezimalwert pro Seite für diesen Drucker
Weitere Parameter
  • public_print_release: Übergeordneter Parameter. Bei „disabled“ können Benutzer ihre eigenen Druckaufträge nicht freigeben, nur Bibliothekspersonal kann dies tun.

3.3.3. Clients konfigurieren

Nachdem Sie Ihren Server für die Druckverwaltung eingerichtet haben, müssen Sie auch Ihre Clients konfigurieren. Dazu müssen Sie die Datei Libki Kiosk Management System.ini bearbeiten. Unter Windows befindet sich diese Datei meist unter C:\ProgramData\Libki, kann sich jedoch je nach Ihrer spezifischen Betriebssystemkonfiguration auch an anderer Stelle befinden. Höchstwahrscheinlich müssen Sie diese Datei als Administrator bearbeiten.

Sobald Sie die Datei gefunden haben, müssen Sie am Ende einen neuen Konfigurationsblock hinzufügen.

[printers]
color="C:\\printers\\color"
monochrome="C:\\printers\\monochrome"

Wie Sie sehen, entsprechen die Drucker den name:-Feldern, die in der Druckverwaltungskonfiguration des Servers definiert wurden. Sie müssen außerdem sicherstellen, dass diese Verzeichnisse auf dem Client-Computer existieren. Sobald der Libki-Client gestartet wurde, überwacht er diese Verzeichnisse auf PDF-Dateien. Sieht der Client eine Datei, lädt er sie mit dem passenden Druckernamen auf den Server hoch.

Libki ermittelt die gewünschte Anzahl der Kopien für einen Druckauftrag anhand des Dateinamens der gespeicherten PDF-Datei. Endet dieser auf das Muster {Zahl}_{Zahl}.pdf, wird die erste Zahl als Kopienparameter interpretiert.

Sie können einen beliebigen PDF-Druckertreiber verwenden, um PDF-Dateien in diese Verzeichnisse zu drucken. Zwei brauchbare, getestete Optionen sind:

  • PDFScribe
  • ClawPDF
Automatische KonfigurationSeit Libki 26.02 kann die Druckerkonfiguration über den Installationsassistenten des Clients vorgenommen werden. Dies setzt voraus, dass ClawPDF als Druckertreiber verwendet wird; dieser muss zuvor installiert werden. Wenn der Installationsassistent des Clients danach fragt, geben Sie die Druckernamen ein, die Sie für den Client aktivieren möchten, eine pro Zeile, wobei die erste Zeile den Standarddrucker darstellt. Der Assistent übernimmt das Anlegen der Druckerverzeichnisse, das Aktualisieren der Client-.ini-Datei und die Konfiguration von ClawPDF.

In diesem Abschnitt können Sie festlegen, was geschieht, wenn sich der Libki-Client abmeldet – entweder weil die Zeit des Benutzers abgelaufen ist oder weil sich der Benutzer vorzeitig abmeldet.

Neustart

Diese Option löst einen vollständigen Neustart des Client-PCs aus. Dies ist nützlich in Kombination mit Software wie Deep Freeze oder Reboot Restore Rx, die den Zustand eines Computers bei jedem Neustart zurücksetzt.

Vom Betriebssystem abmelden

Diese Option löst eine Abmeldung des Windows-Kontos aus, unter dem der Client aktuell angemeldet ist. Dies ist nützlich in Kombination mit der Software Clean Slate, die den Zustand eines Computers bei jeder Abmeldung des Windows-Kontos zurücksetzt.

Nichts

Diese Option belässt das Host-Betriebssystem im aktuellen Zustand und zeigt lediglich den Anmeldebildschirm erneut an. Diese Option wird für den produktiven Einsatz nicht empfohlen. Alle Anwendungen, die der Benutzer zum Zeitpunkt der Abmeldung des Clients geöffnet hatte, bleiben geöffnet. Diese Option ist für Tests und Fehlersuche nützlich.

5.3. Konfigurationsoptionen

Der Libki-Client lässt sich mit Optionen in der Konfigurations-INI-Datei weiter anpassen, die im Installationsassistenten nicht angezeigt werden.

5.3.1. Beschriftungen

Um die Beschriftungen auf dem Libki-Anmeldebildschirm zu ändern, können Sie einen [label]-Block festlegen. Gültige Optionen sind:

  • username (Benutzername)
  • password (Passwort)
  • waiting_holds (wartende Vormerkungen)
  • vom Server empfangene Fehlercodes

Sie müssen nur für die Beschriftungen, die Sie ändern möchten, Zeilen zum [label]-Block hinzufügen. Alle nicht neu definierten Beschriftungen verwenden das Standardwort bzw. die Standardwörter der jeweils genutzten Sprache.

[labels]
username="Username:"                        ; What text it will say at the username input field
password="Password:"                        ; What text it will say at the password input field
waiting_holds="You have holds waiting."     ; This can be used if your client connects to your library's ILS via SIP2.
INVALID_USER="Login Failed: Username and password do not match" ; What text to show on the login screen when this
                                            ; error code is received from the server (any error code can be used)
Übersetzung der Beschriftungen

Um sprachspezifisch benutzerdefinierten Text zuzuweisen, können Sie für jede spezifische unterstützte Sprache einen eigenen [label]-Block definieren. Die Syntax für den neuen Block lautet [labels-<lang>], wobei <lang> der Sprachcode mit oder ohne Länderkennung ist. Beispiel: [labels-en] oder [labels-fr_CA].

[labels-fr_CA]
username="Username:"                        ; What text it will say at the username input field
password="Password:"                        ; What text it will say at the password input field

Beschriftungen in [label] werden als Standardwerte verwendet, wenn im Sprachblock keine Beschriftung definiert ist oder der Block selbst nicht existiert.

5.3.2. Anmeldung ohne Passwort

Wenn Sie Single Sign-on über SIP verwenden und Ihr SIP-Server so eingestellt ist, dass jedes vom Benutzer angegebene Passwort als gültig markiert wird, können Sie den Libki-Client in den passwortlosen Modus versetzen.

Fügen Sie dazu einfach den Schlüssel no_passwords unter dem Abschnitt [node] wie folgt hinzu oder ändern Sie ihn:

no_passwords=1 ; Lets you hide the password field if passwords are not used.

5.3.3. Nutzung des Clients nach Alter einschränken

Es ist möglich festzulegen, dass ein bestimmter Libki-Client nur von Personen einer bestimmten Altersgruppe genutzt werden kann. Um diese Funktion zu nutzen, fügen Sie im Abschnitt [node] der Libki-Client-INI-Datei einen neuen Schlüssel namens age_limit hinzu. Diese Funktion funktioniert nur bei Single Sign-on über SIP, wobei der SIP-Server ein Feld PB im Datumsformat JJJJMMTT zurückgibt, welches das Geburtsdatum des Benutzers darstellt.

Beispiel:

[node]
age_limit="gt18"

Dies beschränkt den Client auf Benutzer über 18 Jahren. Derzeit funktioniert dies nur über SIP2, da es momentan keine Möglichkeit gibt, das Alter eines Benutzers über die Verwaltungsoberfläche zu bearbeiten.

Es können mehrere, durch Kommata getrennte Altersgrenzen angegeben werden, z. B. age_limit="ge11,le17", wodurch der Client auf Benutzer zwischen (einschließlich) 11 und 17 Jahren beschränkt wird.

Die ersten beiden Zeichen des Formats geben den Vergleich an. Unterstützte Vergleiche sind:

  • eq: gleich
  • ne: ungleich
  • lt: kleiner als
  • gt: größer als
  • le: kleiner oder gleich
  • ge: größer oder gleich

Es ist möglich, einen Client für niemanden nutzbar zu machen (z. B. „gt18,lt17“), gehen Sie daher mit dieser Konfiguration vorsichtig um.

5.3.4. Automatische Abmeldung wegen Inaktivität

Es ist möglich, den Libki-Client so zu konfigurieren, dass er einen Benutzer abmeldet, wenn über eine bestimmte Anzahl an Minuten keine Mausbewegung erkannt wurde. Zusätzlich kann der Benutzer nach einer anderen Anzahl von Minuten zunächst gewarnt werden. Beide Konfigurationen existieren als Systemeinstellungen auf dem Libki-Server.

Es ist ebenfalls möglich, diese Konfigurationen pro Client in der Libki-Client-INI-Datei anzugeben. Existieren diese Einstellungen in der Client-INI-Datei, überschreiben sie die Servereinstellungen. Auf diese Weise können die Servereinstellungen als Standard dienen, wobei einzelne Clients diese Servereinstellungen überschreiben können.

Beispiel:

[node]
inactivityLogout=5
inactivityWarning=3

Im obigen Beispiel zeigt Libki nach 3 Minuten ohne Mausbewegung eine Inaktivitätswarnung an. Der Client meldet den Benutzer anschließend nach weiteren 2 Minuten Inaktivität ab. Wird keine Warnung gewünscht, setzen Sie inactivityWarning einfach auf eine höhere Zahl als inactivityLogout.

5.3.5. Externes Programm bei Anmeldung ausführen

Es ist möglich, den Libki-Client so zu konfigurieren, dass er ein weiteres Programm startet, sobald sich ein Benutzer erfolgreich bei Libki anmeldet.

Beispiel:

[node]
run_on_login="C:\\Program Files\\Internet Explorer\\iexplore.exe"

Im obigen Beispiel startet Libki den Internet Explorer, nachdem sich der Benutzer erfolgreich angemeldet hat.

HinweisSie können einfache Schrägstriche anstelle der doppelten Backslashes verwenden (z. B. C:/Program Files/Internet Explorer/iexplore.exe).

6. Sicherung & Wiederherstellung

WichtigEs wird DRINGEND empfohlen, Sicherungen und Wiederherstellungen der Datenbank nur dann durchzuführen, wenn das System gerade nicht genutzt wird!

6.1. Datenbank sichern

Das Sicherungsprogramm ist dafür vorgesehen, auf einem Server mit einem Libki-Benutzer verwendet zu werden.

Im Home-Verzeichnis des Libki-Benutzers wird ein Sicherungsordner angelegt, in dem die Sicherungen gespeichert werden.

Bitte beachten Sie, dass dadurch nur lokale Sicherungen erstellt werden. Es wird empfohlen, diese an einen sicheren Ort zu übertragen. Eine Anleitung zur automatischen Übertragung von Sicherungen auf einen FTP-Server wird später ergänzt.

6.1.1. Manuelle Sicherung erstellen

Führen Sie einfach libki-backup aus.

Die Sicherung wird erstellt und unter /home/libki/backups gespeichert. Der Dateiname enthält Datum und Uhrzeit der Sicherung zur leichteren Identifizierung, z. B. db_180706_19.30.sql.

6.1.2. Automatische Sicherungen erstellen

Das Sicherungsprogramm verfügt über einen stillen Modus, ideal für den Fall, dass Sie keine Benutzereingabe oder Textausgabe wünschen. Führen Sie libki-backup -s aus, um es im stillen Modus auszuführen.

Automatische Sicherungen funktionieren genauso wie manuelle Sicherungen, geben jedoch keinen Text aus. Die Sicherungsdatei wird direkt in /home/libki/backups erstellt.

Wenn Sie Sicherungen automatisch nach einem Zeitplan erstellen möchten, geht dies am einfachsten über crontab. Ich werde nicht im Detail erklären, wie crontab funktioniert, gebe aber einige Beispielzeitpläne an.

Beginnen Sie mit crontab -e. Sie werden wahrscheinlich den Standard-Editor verwenden wollen. Fügen Sie den Zeitplan am Ende der Datei hinzu.

Crontab-Beispiele

Tägliche Sicherung um 02:00 Uhr:

0 2 * * * /usr/local/bin/libki-backup -s

Sicherung jeden Sonntagabend:

0 21 * * SUN /usr/local/bin/libki-backup -s

Sicherung am ersten Tag jedes Monats um 01:15 Uhr:

15 1 1 * * /usr/local/bin/libki-backup -s

6.2. Serverdateien sichern

Der einfachste Weg, die Serverdateien zu sichern, besteht darin, den Ordner libki-server im Home-Verzeichnis des Libki-Benutzers zu tarn. Es empfiehlt sich, diesen im selben Sicherungsordner wie die Datenbanksicherung abzulegen.

tar -pzcvf /home/libki/backups/libki-server-backup_190816.tar.gz /home/libki/libki-server

6.3. Datenbanksicherung wiederherstellen

Das Wiederherstellen einer Datenbanksicherung löscht die aktuelle Datenbank; stellen Sie also sicher, dass Sie dies wirklich tun möchten.

Wechseln Sie zunächst in den Ordner, der die wiederherzustellende Sicherung enthält. Wurde die Sicherung mit dem Sicherungsprogramm erstellt, ist dieser Ordner /home/libki/backups.

cd /home/libki/backups

Führen Sie nun das Programm aus und geben Sie an, welche Sicherung Sie wiederherstellen möchten.

libki-restore db_180706_19.30.sql

Ihre Datenbank befindet sich nun in dem Zustand, in dem sie sich zum Zeitpunkt der Sicherung befand.

6.4. Sicherung der Serverdateien wiederherstellen

Entfernen Sie zunächst die aktuellen Serverdateien und ersetzen Sie sie durch das Sicherungsarchiv.

service libki stop
rm -rf /home/libki/libki-server
tar -pzxvf /home/libki/backups/libki-server-backup_190816.tar.gz /home/libki/libki-server
service libki start

7. Server aktualisieren

7.1. Vorsichtsmaßnahmen

Es wird DRINGEND empfohlen, den Server nur dann zu aktualisieren, wenn das System gerade nicht genutzt wird und der Dienst heruntergefahren ist.

service libki stop

Erstellen Sie anschließend eine Sicherung Ihres aktuellen Servers und Ihrer Datenbank, falls etwas nicht so reibungslos verläuft wie geplant. Die Aktualisierung ist nicht in jedem Fall vollständig stabil, insbesondere bei der Aktualisierung von sehr alten Installationen.

Bitte lesen Sie den Abschnitt zur Sicherung durch, um zu erfahren, wie Sie Ihren Server sichern.

7.2. Neueste Version holen und Datenbank aktualisieren

Wechseln Sie in Ihren libki-server-Ordner und verwenden Sie git pull, um die neueste Version herunterzuladen. Führen Sie installer/update_db.pl aus, um die Datenbank zu aktualisieren.

cd libki-server
git pull
./installer/update_db.pl

ACHTUNG: Wenn Sie von einer 2.x-Installation aktualisieren, funktioniert dies nicht. Sie müssen zunächst die neueste 2.x-Version r19.08 als Zwischenschritt verwenden, bevor Sie auf einen 3.x-Server aktualisieren, um die Datenbank zu konvertieren.

cd ~
apt update && apt install unzip -y
wget https://github.com/Libki/libki-server/archive/r19.08.zip
unzip r19.08.zip
./r19.08/installer/update_db.pl

Jetzt können Sie erneut das aktuelle update_db.pl ausführen.

7.3. Serverdateien aktualisieren

Sobald die Datenbank auf dem neuesten Stand ist, müssen die Serverdateien aktualisiert werden. Löschen Sie die alten Serverdateien, ersetzen Sie sie durch die neuen und übertragen Sie deren Besitz an den Libki-Benutzer.

cd ~
cp /home/libki/libki-server/libki_local.conf ./
cp /home/libki/libki-server/log4perl.conf ./
rm -rf /home/libki/libki-server
cp -r libki-server /home/libki/libki-server
cp ./libki_local.conf /home/libki/libki-server/
cp ./log4perl.conf /home/libki/libki-server/
chown -R libki:libki /home/libki/libki-server

Sobald dies erledigt ist, ist es Zeit, den Dienst wieder zu starten. Ist alles ordnungsgemäß verlaufen, sollte Ihr Libki-Server nun aktualisiert sein und genauso gut (oder sogar besser) funktionieren als zuvor.

service libki start

8.2. Übersetzung

8.2.1. Den Server übersetzen

Eine neue Übersetzung beginnen

Der Server wird über eine .po-Datei übersetzt, die sich in lib/Libki/I18N befindet. Die .po-Datei basiert auf einer Vorlage im selben Ordner namens messages.pot. Sie wird dynamisch aus dem Server generiert und anschließend mit einem eigenen Wörterbuch kombiniert.

Bevor Sie etwas anderes tun, müssen wir die Sprache, in die wir übersetzen, zum Wörterbuch hinzufügen.

cd lib/Libki/I18N
nano extras.pot

Der erste Teil dieses Wörterbuchs ist der wichtige Teil. Fügen Sie Ihre Sprache zum Sprachmenü hinzu. Folgen Sie derselben Syntax, die auch die anderen Sprachen verwenden, und ordnen Sie sie alphabetisch ein. Hier füge ich Dänisch hinzu, das würde also ganz oben stehen. Eine vollständige Liste der Sprachcodes finden Sie hier.

# Language menu

# Danish
msgid "lang.da"
msgstr "Dansk"

# English
msgid "lang.en"
msgstr "English"

Sie müssen dies auch in allen anderen Sprachdateien ergänzen. Da diese zu unterschiedlichen Zeiten verfasst wurden, kann sich die Stelle am Anfang oder am Ende der Datei befinden.

Aktualisieren wir nun messages.pot, indem wir eine neue Version generieren. Sie sollte immer dann neu generiert werden, wenn an einer Übersetzung gearbeitet wird. Um eine neue Version zu generieren, stellen Sie sicher, dass libki-server Ihr aktuelles Verzeichnis ist.

./script/utilities/translate.sh --generate

Kopieren Sie die Vorlage in Ihre Sprache und bearbeiten Sie sie.

cp lib/Libki/I18N/messages.pot da.po

Es wird dringend empfohlen, einen dedizierten .po-Editor zu verwenden. Wir empfehlen Poedit.

Eine bestehende Übersetzung aktualisieren

Wenn Sie eine bestehende Übersetzung aktualisieren möchten, müssen Sie eine neue messages.pot generieren und Ihre Übersetzung so aktualisieren, dass sie auf der neuen messages.pot basiert. Im Rahmen des Aktualisierungsprozesses werden geänderte Inhalte markiert, damit sie leicht zu finden sind.

Generieren Sie zunächst eine neue messages.pot. Stellen Sie sicher, dass libki-server Ihr aktuelles Verzeichnis ist.

./script/utilities/translate.sh --generate

Sobald diese generiert ist, aktualisieren Sie die bestehende Übersetzung. Auch dies muss aus libki-server heraus ausgeführt werden. Wir verwenden hier Dänisch als Beispielübersetzung.

./script/utilities/translate.sh --update lib/Libki/I18N/da.po

Eine Sicherungskopie wird in lib/Libki/I18N erstellt, falls Sie sie benötigen. Denken Sie nur daran, sie vor dem Erstellen eines Pull Requests zu löschen.

Wenn Sie Poedit zur Bearbeitung Ihrer Übersetzung verwenden, lässt sich die nächste unvollständige Zeile leicht mit Umschalt-Strg-Pfeil-nach-unten finden.

8.2.2. Den Client übersetzen

Der Client wird mit einer .qm-Datei übersetzt, die aus einer .ts-Datei generiert wird. Wir bearbeiten die .ts-Datei, und sobald das erledigt ist, generieren wir eine .qm-Datei. Sie benötigen Qt Linguist. Unter Linux ist das qt5-dev-tools: sudo apt install qt5-dev-tools. Unter Windows lädt man es am einfachsten von Thurasks GitHub herunter: https://github.com/thurask/Qt-Linguist/releases

Zuerst müssen wir Libki.pro bearbeiten. Öffnen Sie die Datei und fügen Sie Ihre Sprache zur Liste der Übersetzungen hinzu. Verwenden Sie dieselbe Syntax wie die anderen Einträge. Hier füge ich Dänisch hinzu. Eine vollständige Liste der Sprachcodes finden Sie hier.

TRANSLATIONS = languages/libkiclient_fr.ts \
        languages/libkiclient_sv.ts \
        languages/libkiclient_da.ts

Als Nächstes wird die .ts-Datei erstellt. Dies erfolgt mit lupdate Libki.pro, entweder im Terminal, in cmd oder in Powershell.

Verwenden Sie nun Qt Linguist, um Ihre .ts-Datei zu übersetzen. Sie befindet sich im Verzeichnis languages.

Ist das erledigt, erstellen Sie eine .qm-Datei mit lrelease Libki.pro. Dadurch werden auch die anderen Sprachdateien aktualisiert.

Fügen Sie schließlich die .qm-Datei zu libki.qrc hinzu. Hier habe ich Dänisch am Ende hinzugefügt.

...
    <qresource prefix="/">
        <file>languages/libkiclient_fr.qm</file>
        <file>languages/libkiclient_sv.qm</file>
        <file>languages/libkiclient_es.qm</file>
    </qresource>
</RCC>
